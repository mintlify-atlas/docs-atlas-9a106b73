## What problem does it solve?

Building multi-agent workflows by hand means writing the same boilerplate every time: parse the goal, decide which agents run first, handle dependencies, collect results, synthesize a final answer. This is error-prone and hard to change when requirements shift.

Open Multi-Agent replaces that boilerplate with a **coordinator pattern**: a built-in coordinator agent receives your high-level goal, breaks it into a typed task DAG with explicit dependencies and assignees, and hands it to the scheduler. Independent tasks run in parallel automatically. You get structured progress events and a typed result — no polling, no custom queues.

```
Goal: "Create a REST API for a todo list in /tmp/todo-api/"
                          │
               ┌──────────▼──────────┐
               │   coordinator       │  ← decomposes into task DAG
               └──────────┬──────────┘
                 ┌─────────┴──────────┐
         ┌───────▼──────┐    ┌────────▼───────┐
         │  architect   │    │  developer (2) │  ← independent, run in parallel
         └───────┬──────┘    └────────┬───────┘
                 └─────────┬──────────┘
               ┌──────────▼──────────┐
               │    reviewer         │  ← unblocked after implementation
               └──────────┬──────────┘
               ┌──────────▼──────────┐
               │   coordinator       │  ← synthesizes final answer
               └─────────────────────┘
```

Four subsystems collaborate on every run:

- **Coordinator** — A temporary agent that decomposes goals into a JSON task list and synthesizes the final answer.
- **TaskQueue / task DAG** — Holds tasks with typed statuses and dependency edges. Automatically unblocks tasks when their dependencies complete.
- **AgentPool** — A semaphore-limited pool that runs agents concurrently up to `maxConcurrency` (default: 5).
- **SharedMemory** — A key-value store that persists each task's result so later agents can read prior work.

## The three run modes

| Mode | Method | When to use |
|------|--------|-------------|
| Single agent | `runAgent(config, prompt)` | One agent, one prompt — simplest entry point for quick tasks |
| Auto-orchestrated team | `runTeam(team, goal)` | Describe a goal; the framework plans, schedules, and executes |
| Explicit pipeline | `runTasks(team, tasks)` | You define the task graph and assignments, framework executes |

## Key features

  <Card title="Parallel-by-default execution" icon="bolt">

## Supported providers

| Provider | `provider` value | Env var | Status |
|----------|-----------------|---------|--------|
| Anthropic (Claude) | `'anthropic'` | `ANTHROPIC_API_KEY` | Verified |
| OpenAI (GPT) | `'openai'` | `OPENAI_API_KEY` | Verified |
| GitHub Copilot | `'copilot'` | `GITHUB_TOKEN` | Verified |
| Ollama / vLLM / LM Studio | `'openai'` + `baseURL` | — | Verified |

<Note>
  Any OpenAI-compatible API works via `provider: 'openai'` and `baseURL`. DeepSeek, Groq, Mistral, Qwen, and others have not been fully verified — contributions welcome via [#25](https://github.com/JackChen-me/open-multi-agent/issues/25).
</Note>

## Built-in tools

Five production-ready tools are registered automatically for every agent that lists them:

| Tool | Description |
|------|-------------|
| `bash` | Execute shell commands. Returns stdout and stderr. Supports timeout and working directory. |
| `file_read` | Read file contents at an absolute path. Supports `offset` and `limit` for large files. |
| `file_write` | Write or create a file. Auto-creates parent directories. |
| `file_edit` | Edit a file by replacing an exact string match. |
| `grep` | Search file contents with regex. Uses ripgrep when available, falls back to Node.js. |

You can define additional tools with `defineTool()` and a Zod schema. See the [custom tools guide](/guides/custom-tools).

## vs. manual orchestration

| Capability | Manual orchestration | Open Multi-Agent |
|-----------|---------------------|-----------------|
| Task decomposition | You write it | Coordinator agent generates it |
| Dependency resolution | You topologically sort | `TaskQueue` handles it |
| Parallel execution | You manage `Promise.all` | `AgentPool` + semaphore |
| Shared context | You pass results by hand | `SharedMemory` auto-injects |
| Progress visibility | You instrument everything | Built-in `OrchestratorEvent` stream |
| Model switching | Different SDK per provider | `provider` + `baseURL` per agent |
